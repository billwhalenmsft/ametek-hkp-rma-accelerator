RMA Operations — Help & Cheat Sheet v5 - 5/20

Quick reference for everyday tasks + full admin reference for the Settings tables. Last updated 2026-05-20.
Jump to: First setup File a new claim Process inbound email Use claim actions Read Smart Insights Request approval Move a claim through stages Find prior claims Saved views Settings Troubleshooting Architecture (devs) Resolution & ERP

1. First setup

  1. Open the Configuration group in the left nav and walk the 5 tables in order: Plants → Routing Rules → Plant Approvers → Email Templates → Email Signatures. Each one builds on the prior, so do them in order on first setup.
  2. Plants: create one row per manufacturing location. Set Region, Part Prefixes (comma-separated, e.g. HKP-,SAN-), Product Lines, and Auto Credit Threshold (the dollar value below which the system auto-approves with no manager).
  3. Routing Rules: tell the system which plant a claim should go to. Pick a Rule Type (Part Prefix / Customer Region / Product Line), enter the Match Value, pick the Assigned Plant, and set a Priority (lower = wins).
  4. Plant Approvers: for each plant, add the people who get the Teams approval card. Use Notify When = All Claims for everyday reviewers (plant manager) and High Value Only + a High Value Threshold (e.g. $5,000) for VP-level escalation. Set Teams UPN to the user's Microsoft 365 sign-in.
  5. Email Templates: author reusable customer reply content. Mark each template with a Type (Submission Confirmation, Status Update, Approval, Denial, Custom) and toggle Active. Templates surface on the claim form's reply buttons.
  6. Email Signatures: create one row per signer; mark exactly one as Is Default. The default signature is auto-appended to every outbound reply.
  7. Once Configuration is done, open Email Inbox to triage inbound messages and use Email Records under History for audit/search.
Tip: Both the Main forms and the Quick Create panels for the Configuration tables now show inline help text in the section header explaining what each table does.
Known gap (5/20): The current Request Manager Approval flow notifies all active approvers for a plant. The Notify When + High Value Threshold fields are honored in the form but not yet in the flow filter — coming next.

2. File a new claim

  1. Click the global + New button (top toolbar) → RMA Claim.
  2. Fill the Quick Create form: customer, part number, quantity, failure mode.
  3. Click Save and Close — a claim number (RMA-…) is auto-assigned.
  4. The auto-assignment flow picks a plant based on part-number prefix.
Tip: If the suggested plant is wrong, change Assigned Plant on the form. The flow re-evaluates routing rules whenever the part number or region changes.

3. Process inbound email

  1. Open Email Inbox from the left nav for the full-page triage experience.
  2. Review the email content and extracted details, then open or create the related claim.
  3. Use the Email Assist experience to draft or send customer replies when needed.
  4. Once processed, the message is tracked in Email Records under History for audit/search.
  5. Open an individual email log record if you need the side pane and full activity details.
Tip: The email scraper deduplicates on Message ID, so re-running ingestion should not create duplicates.

4. Use claim actions from the Pizza Tracker

  1. Open any claim and look at the Pizza Tracker at the top of the form.
  2. Use the inline action buttons there for the most common tasks instead of relying on the top command bar.
  3. Submission Confirmation sends the customer acknowledgement template.
  4. Status Update sends the current-status template using the claim/customer context.
  5. Resolve, Deny, and Request Approval actions are also available inline for faster claim handling.
Tip: The current template merge supports first-name style greetings for customer-facing messages, so you should no longer see the full name dropped into greeting-only spots.

5. Read the Smart Insights panel

The Smart Insights panel (right side of the claim form) summarizes what the system thinks about the claim:

Customer Health0–100 score, derived from prior-claim volume, denial rate, average resolution time Part Risk0–100 score, derived from defect rate across all customers and warranty period RecommendationAuto-Approve / Investigate / Escalate Customer HistoryTop 5 prior claims for the same customer — click any to open
Note: The Navision integration is currently stubbed (deterministic fake) and will be replaced with the live ERP feed. Behavior + UI won't change.

6. Request manager approval

  1. Open the claim, fill in the Credit Amount (or other resolution amount).
  2. Click Request Manager Approval on the Pizza Tracker (or move status to Decision). The RMA: Request Manager Approval for Claim flow fires.
  3. The flow looks up all active Plant Approvers for the claim's assigned plant and sends each one an Adaptive Card in Teams.
  4. The first approver to click Approve or Deny wins — the claim's Status and Resolution are updated automatically, a row is written to Approval History, and a customer email is sent if the matching template is active.

Two-tier approval model (configured per approver in Plant Approvers):

Notify When = All ClaimsApprover gets every claim for their plant. Typical for the day-to-day plant manager. Notify When = High Value OnlyApprover only gets the card when the claim's resolution amount is ≥ their High Value Threshold. Typical for VP-level escalation (e.g. $10,000+). Notify When = Manual OnlyApprover never gets auto-notified; included only when explicitly added by another approver.
Known gap (5/20): The current flow ignores Notify When and High Value Threshold and notifies all active approvers for the plant. Dollar-tier filtering is the next planned flow update. Until then, model the tiers but expect everyone to be pinged.
Auto-approve below threshold: If the resolution amount is below the plant's Auto Credit Threshold, the claim auto-approves with no manager card sent.

7. Move a claim through stages

The claim status drives the Pizza Tracker visualizer at the top of the form. Allowed stages:

  • New — just filed; auto-assignment is still running
  • Triage — owner reviewing; deciding investigation path
  • Investigation — engineering / quality looking at part
  • Decision — approver review; resolution amount set
  • Closed — final disposition recorded; customer notified

To advance the stage: edit Status in the form header. The Pizza Tracker should refresh to match the saved stage, and the "stage age" pill turns red after 7 days in the same stage.

8. Find prior claims for a customer or part

Three ways:

  1. Quick Find on the RMA Claims grid (top-right magnifier) searches across customer, part, and claim number.
  2. The Smart Insights panel on any open claim shows the top 5 prior claims for that customer.
  3. Claims Board dashboard supports inline filter by part number prefix.

9. Saved views to know

  • Active Claims — default open queue
  • Awaiting My Action — assigned to you, in Triage or Decision
  • Stuck (>7 days) — anything that hasn't moved a stage in a week
  • By Plant — grouped by Assigned Plant
  • By Status — grouped by stage
  • By Customer Region — grouped by region
  • High-Value — credit amount > $5,000
  • Closed This Week
  • Closed This Month

Routing Rules

Tells the system which plant a new claim should be assigned to. Multiple rules evaluate in priority order; first match wins.

Rule NameHuman label, e.g. HKP parts → Sanford Rule TypePart Prefix (match part number prefix) / Customer Region (NA/EMEA/etc.) / Product Line Match ValueThe literal string to match. Examples: HKP- for Part Prefix, EMEA for Region, Pumps for Product Line. Case-insensitive prefix match. PriorityTiebreaker when multiple rules match. Lower wins. Example rule set:
• HKP parts to Sanford — Part Prefix / HKP- / Priority 10 / HKP-Sanford
• SAN parts to Mexico — Part Prefix / SAN- / Priority 10 / HLM-Mexico
• EMEA fallback — Customer Region / EMEA / Priority 50 / HLM-Italy
• China overflow — Customer Region / CN / Priority 50 / HLM-China

Add a new lookup relationshipUse Add-Lookup in the create script. After running, register a sub-grid on the parent form via scripts/update_config_table_forms.ps1 (clone its New-Row pattern). 12.3 Power Automate flows — design + how to modify

All flows live in the RMAReturnsMonitor solution. Each is deployed by a clone-script that POSTs a fully-formed clientdata JSON to /api/data/v9.2/workflows, then activates the workflow. After re-running any clone script you MUST open the flow in the modern designer and click Save once — that single action registers the Dataverse trigger webhook (programmatic POST does not).

Request Manager Approval for ClaimTrigger: rma_approvalrecord row added. Actions: Get parent rma_claim → list active plant approvers for that plant honoring rma_notifywhen + rma_highvaluethreshold → send Teams Approval card → on approve: set approval Approved + close claim + write history; on deny: set approval Denied + leave at Decision + write history. Source: scripts/clone_request_manager_approval_for_rma.ps1. Modify the approver filter: edit the List_active_plant_approvers action's $filter expression in the clone script. RMA Email MonitorTrigger: shared mailbox new email. Actions: upsert into rma_emailinbox, attach blobs, link to existing claim if subject matches a claim number. Source: deployed manually via solution import — see solution/ folder. Auto-classify EmailTrigger: rma_emailinbox row added. Actions: call AI Builder model to extract claim type + plant hint, then walk rma_routingrule rows in priority order to set rma_classification + suggested rma_plant. Process Approval DecisionTrigger: rma_approvalrecord rma_approvalstatus changes to Approved or Denied (covers Teams approvals AND manual override from the Pizza Tracker). Actions: update parent rma_claim, append rma_approvalhistory row. Send Templated EmailTrigger: rma_claim rma_status changes. Actions: look up matching rma_emailtemplate by status transition, substitute placeholders ({customer_name}, {claim_number}, {signature}, etc.) using the customer's email + the default rma_emailsignature, send via Office 365 connector.
Gotchas (codified in CLAUDE.md + user memory):
• Trigger webhook is registered by the maker UI on first Save — never by REST POST alone.
• If you change a flow's response schema, the bound topic / caller's binding cache is stale — delete + re-add the calling action (cannot be fixed via REST).
• Connection references must already exist in the env. New flows reuse cr74e_sharedapprovals_453c9 and the shared CDS connection ref.

13. Resolution workflow & ERP integration

How the Issue Credit / Send Replacement / Repair Completed / Other Action buttons work end-to-end, what fields they write, and how to swap the Navision stub for a real ERP call.

13.1 What happens when you click a resolve button

  1. Modal opens with a template picker (filtered by action: Credit:, Replacement:, Repair:, or all non-denial for Other), an editable email body (placeholders auto-merged from the claim), and an editable amount field (Credit/Replacement only).
  2. Amount default: the most recent Approved rma_approvalrecord for this claim, otherwise the claim's rma_creditamount. If you change it, an Override reason field appears and is required.
  3. On Confirm three things happen in parallel:
    • A timeline note is written (subject + full email body + override info).
    • An rma_approvalhistory row is created with: rma_action, rma_amount, rma_amountoriginal, rma_amountoverridden, rma_overridereason, rma_emailto, rma_emailbody, rma_erpstatus (Pending for Credit/Replacement, Not Required for Repair/Other).
    • If "Send email" is checked, a Dataverse email activity is created and sent to the customer (regarding the claim).
  4. Claim is closed: rma_resolution set, rma_status → Closed, rma_closeddate stamped.
  5. If rma_erpstatus = Pending, the RMA: Push Resolution to ERP (stub) flow fires within ~30s, generates a stub reference (STUB-NAV-xxxxxxxxxxxx), and flips the history row to Sent.

13.2 Schema additions (rma_approvalhistory)

rma_amountMoney — what was actually paid/issued. rma_amountoriginalMoney — what the manager originally approved (for audit diff). rma_amountoverriddenBool — true if operator changed amount at confirm. rma_overridereasonText 500 — reason the operator gave for the override. rma_emailtoText 200 — recipient of the resolution email. rma_emailbodyMemo — snapshot of the body that was sent (as the operator edited it). rma_erpstatusPicklist: Not Required / Pending / Sent / Failed. rma_erpreferenceText 100 — doc id returned by ERP (or stub ref). rma_erppayloadMemo — full JSON pushed to ERP, for audit/debug. rma_action (new options)Credit Issued (100000004), Replacement Sent (100000005), Repair Completed (100000006), Other Action (100000007).

13.3 The ERP stub flow

Source: scripts/clone_push_resolution_to_erp.ps1. Trigger: rma_approvalhistory row added with rma_erpstatus = Pending. Actions: get parent claim → build payload (Compose) → STUB ref generator (Compose) → PATCH history row to Sent.

Important: After running the script, you must open the flow in Power Automate maker UI and click Save once. Programmatic POST creates the flow shell but does not register the Dataverse trigger webhook — the runtime registers it on first maker-UI save.

13.4 Swapping the stub for a real Navision push

  1. Open the flow in maker UI. Find the action Push_to_ERP_STUB.
  2. Delete it. Add an HTTP action in its place named Push_to_ERP:
    • Method: POST
    • URI: your Business Central / Navision endpoint (likely via on-prem data gateway)
    • Headers: Authorization: Bearer ... (use a connection reference for OAuth/Azure AD)
    • Body: @{outputs('Build_ERP_payload')}
  3. In the next action Update_history_row_Sent, bind rma_erpreference to the doc id returned by the HTTP call (e.g. @{body('Push_to_ERP')?['documentId']}).
  4. Add a Configure run after on the update action: also run when Push_to_ERP fails, and in that branch PATCH rma_erpstatus → Failed with the error message in rma_erppayload.
  5. Save and re-test.
