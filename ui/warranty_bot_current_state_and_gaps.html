1. TL;DR

The Warranty Claim Processing for Email bot in Master CE Mfg today does an excellent intake-and-acknowledgment workflow. It parses inbound emails, asks the customer for missing fields via HTML email templates, creates a warranty claim record in Dataverse, and posts to Teams. 5/5 intake actions wired

What it doesn't do yet — and what HKP RMA needs — is triage decisions: warranty lookup by serial, eligibility check against rules, disposition recommendation with confidence + rationale, escalation routing to a Quality Engineer. 0/4 triage capabilities

The good news: this is a 4-step extension to the agent's instructions: block + 2 new actions. We don't replace, we extend. The intake flow stays exactly as built.

Net effort estimate: ~1 dev day to add the 4 instruction steps + 2 actions + test against 5 sample claims. Then a follow-up session with HKP to calibrate failure-mode keywords, OEM tier mapping, and engineer routing.

2. What's in the bot today

The 5-step flow (encoded in instructions:)

This bot is a generative agent — almost all behavior lives in the instructions block, not in topics. When the email-arrival trigger fires, the agent reads the instructions and orchestrates calls to the 5 actions in this order:

1Extract entities from inbound email → uses ExtractEntitiesfromEmailResponseV2 action. Pulls FirstName, LastName, SerialNumber, Phone, Email, ProblemDescription.

2If fields missing → reply with HTML template asking customer to fill in the gaps. Uses Office365Outlook-SendanemailV2. (Email template embedded in instructions — formatted HTML table with First Name / Last Name / Serial / Phone / Email / Problem Description rows.)

3On reply, re-extract. If complete → CreateWarrantyClaim action writes the record to cr74e_warrantyclaim. If still incomplete → loop back to step 2.

4Send confirmation email to customer with claim # → Office365Outlook-SendanemailV2_T1N action.

5Post Teams card notifying Warranty Management Team / Contoso Warranty channel → MicrosoftTeams-Postcardinachatorchannel action.

Architectural note: This is the "single LLM-orchestrated agent" pattern, not the "router with N topics" pattern. That's actually ideal for adding triage — we extend the instructions block with steps 6-9, the same agent reads them.

3. What HKP RMA needs (the gap)

Extend the agent's instructions block — 4 new steps

After current step 5 (Teams card), add:

6Look up warranty status for the serial number. Returns: SKU, owner, shipped date, warranty end, in-warranty boolean, days remaining. NEW ACTION needed

7Apply eligibility rules:
• In warranty + manufacturing-defect keyword → eligible
• Out of warranty by >90 days → ineligible
• Borderline / EMI suspected / sparse description → unclear
• Misuse keywords (water, impact, lightning, etc.) → ineligible

8Recommend disposition with confidence + rationale. One of: approve_repair / approve_replace / approve_refund / reject / request_more_info / escalate_to_engineer. Always include the supporting facts list.

9Branch on disposition:
• If approve_replace → trigger Send Shipping Label flow + send customer email
• If reject → send customer rejection email with paid-repair quote offer
• If request_more_info → send customer email asking for photos + error codes
• If escalate_to_engineer → call new RouteEscalation action (assigns QE based on product family, posts evidence pack to engineer's Teams DM)
• If confidence < 0.70 → also trigger Request Manager Approval for Claim flow even on auto-decisions

4. Build plan (3 phases)

Phase 1 — Validate today (30 min)

Goal: confirm the current bot still works end-to-end before we change anything.

  1. Send a test email to the bot's monitored inbox with a serial + problem description
  2. Verify: bot extracts entities, replies if missing fields, creates the claim, sends confirmation, posts Teams card
  3. Capture screenshot of: the email reply HTML, the Dataverse claim record, the Teams adaptive card
  4. Status check via Power Automate run history — all 5 action calls succeed
Output: 3 screenshots that go into the customer call as "here's the foundation we're building on."

Phase 2 — Extend the bot (~1 dev day)

Goal: bot does intake-through-disposition end-to-end.

  1. Build the 2 new Power Automate flows: LookupWarrantyBySerial + RouteEscalation (~3 hours)
  2. Add 4 instruction steps to agent.mcs.yml (steps 6-9 above)
  3. Activate the 2 currently-OFF flows: Claim Routing Automation + Capture each incoming and outgoing email
  4. Add 6 customer-specific config JSON files under customers/ametek/hkp_rma/config/
  5. Push via VS Code Copilot Studio extension as draft (doesn't change live bot)
  6. Test in CS test pane against 5 sample emails (mirror the HKP RMA tester's 5 scenarios)
  7. If pass → Publish in CS UI
Risk: First push may fail with binding cache or schema strictness errors. See /memories/power-platform-deploy.md in the repo for the documented fixes (delete-and-readd Action node, etc.). Budget 1 hour for push iteration.

Phase 3 — Productionize (after Phase 2 demo passes)

  1. Calibrate failure-mode keywords + disposition rationale to HKP vocabulary (from customer call notes)
  2. Replace Milwaukee KB with HKP/AMETEK warranty terms doc
  3. Replace mock engineer roster with real names + Teams IDs
  4. Wire historical-context lookup (read closed claims for similar failures, append to disposition rationale)
  5. Add acceptance test harness (~5 cases × 4 dispositions = 20 tests) running in CI on draft bot
  6. Promote from Master CE Mfg → customer's actual env (or keep customer in Master CE if it's their env)
  7. Hand off to admin team with runbook + rollback procedure

5. What's generic vs customer-specific (template strategy)
